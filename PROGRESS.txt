PHASE 1 — Pixel half: fixed distortion, duration, smoothness  [DONE]
  Files: Recorder.java, RecorderPlayer.java, RecorderExporter.java
  - DISTORTION: capture now reads the framebuffer the same way Minecraft's own
    Screenshot does (native ints, vertical flip, force alpha) and stores
    v3 frames; playback decodes via NativeImage.setPixelABGR + texture.upload().
    Verified end-to-end against MC source (writeToTexture does no Y-flip; blit
    maps texture-top -> screen-top) -> provably upright.
  - DURATION: playback advances by real per-frame TIMESTAMPS (frameTimestamps[])
    instead of an assumed fps; on-screen timer (fmt) also uses timestamps, so a
    25s clip shows 0:25 not 0:04.
  - SMOOTHNESS: root cause of ~10fps was ~24MB per-frame allocations on the
    RENDER thread (GC thrash). Fixed by:
      * capture does only mapBuffer + pooled memcpy on render thread (ZERO
        per-frame allocation there),
      * vertical-flip + deflate moved entirely to the writer thread,
      * raw GPU buffers pooled and reused.
  - Container version bumped 2 -> 3 (old broken files not mis-decoded).
  - RecorderExporter updated: skips the vflip filter for v3 (capture already
    flips) so MP4 export isn't double-flipped.

PHASE 2 — State half (recording): packets + ticks + meta  [DONE, compiles/builds]
  Files: StateRecorder.java (new), ConnectionMixin.java (new),
         ClientTickMixin.java (new), Recorder.java (wired),
         crest-client.mixins.json (registered)
  - Records gameplay as STATE into a sidecar file "<name>.crest.state"
    (magic "CRS1"), as timestamped records:
      * META   — seed(placeholder 0), dimension, player UUID, spawn, w/h
      * PKT_IN / PKT_OUT — every gameplay packet, encoded via
        GameProtocols.CLIENTBOUND_TEMPLATE / SERVERBOUND_TEMPLATE codecs,
        using the LIVE RegistryAccess so entity/block/registry packets serialize.
      * TICK   — one marker per client tick.
  - Mixins: Connection.channelRead0 (inbound) + Connection.send (outbound),
    gated to in-world PLAY phase; Minecraft.tick() for tick markers + first-tick
    META.
  - Fully defensive: every interception wrapped in try/catch and gated on
    level!=null && player!=null. A packet that fails to encode is skipped — it
    can NEVER crash the game or disturb the pixel recorder.
  - Wired into Recorder.start()/stop() so state + pixel recording run together.
  - Full `./gradlew build` (compile + remap + package) SUCCEEDS.

API research done (MC 26.1.2 specifics):
  - ResourceLocation renamed to Identifier (net.minecraft.resources.Identifier).
  - SimpleUnboundProtocol (clientbound) vs UnboundProtocol<_,_,Context>
    (serverbound) — different template types, handled separately.
  - Level has no getSeed() on client -> placeholder 0 (read from live world in
    Phase 3).
  - ConnectionProtocol enums have no direct codec accessor; use
    GameProtocols.CLIENTBOUND_TEMPLATE / SERVERBOUND_TEMPLATE.bind(...).
  - ClientPacketListener.registryAccess() for live registry during encode.

PHASE 3 — Playback re-simulation + input capture  [DONE, compiles/builds]
   Files: StatePlayer.java (new), ReplayTickMixin.java (new),
          ConnectionBlockMixin.java (new), MouseHandlerAccessor.java (new),
          StatePlayerHolder.java (new), MouseInputMixin.java (new),
          KeyboardHandlerMixin.java (new), StateRecorder.java (extended),
          crest-client.mixins.json (updated)
   - INPUT capture: KeyboardHandlerMixin records keyPress(win,key,scancode,action,mods)
     as TYPE_KEY records. MouseInputMixin records onPress → TYPE_MOUSE_BTN,
     onScroll → TYPE_MOUSE_SCROLL, turnPlayer → TYPE_MOUSE_MOVE.
     All gated to in-world (level!=null, screen==null), wrapped in try/catch.
   - StateRecorder extended with 4 new record types (KEY, MOUSE_BTN, MOUSE_SCROLL,
     MOUSE_MOVE) and build/send helpers using FriendlyByteBuf encoding.
   - StatePlayer: reads .crest.state (magic "CRS1"), parses all records into typed
     list (Meta, PKT_IN, PKT_OUT, TICK, KEY, MOUSE_BTN, MOUSE_SCROLL, MOUSE_MOVE),
     provides pollUpTo(timeUs) for replay, and decodes packets via the same
     GameProtocols codecs used for encoding.
   - ReplayTickMixin: injects at Minecraft.tick HEAD. When StatePlayerHolder.active
     is set, it polls due records up to elapsed replay time. PKT_IN decoded and
     fed into ClientPacketListener via raw-cast handle(). MOUSE_MOVE sets
     accumulatedDX/DY via MouseHandlerAccessor then calls handleAccumulatedMovement().
     KEY replay uses lambda$setup$4 reflection trampoline.
   - ConnectionBlockMixin: suppresses Connection.tick() during replay to prevent
     double-feeding (real connection packets + injected replay packets).
   - MouseHandlerAccessor: exposes accumulatedDX/accumulatedDY for replay rotation.
   - StatePlayerHolder: singleton bridge (active, startTimeMs, elapsedUs/durationUs).
   - Full `./gradlew build` SUCCEEDS.
   NOTES:
     - KEY replay uses reflection trampoline (lambda$setup$4); may need refinement
       if the lambda signature differs across MC versions.
     - MOUSE_MOVE sets both accumulatedDX + accumulatedDY in one call then processes
       both via handleAccumulatedMovement(). Verified against MC 26.1.2 decompiled
       MouseHandler API (accumulatedDX/DY are private double fields, turnPlayer(double)
       is single-arg, handleAccumulatedMovement() is public).
     - PKT_IN injection uses unchecked Packet<PacketListener> cast matching
       Connection.genericsFtw pattern (same cast the real Connection does).
     - Minecraft.getConnection() returns ClientPacketListener (not Connection!) in
       MC 26.1.2. Adjusted all code to use this API.
   KNOWN LIMITATIONS (MVP):
     - Re-simulation does NOT yet create a synthetic world — you must replay while
       connected to a world (existing singleplayer/multiplayer). The mixin feeds
       recorded packets into the live connection; world state updates accordingly.
       Full world-from-scratch replay is Phase 4 stretch.
     - MOUSE_BTN/MOUSE_SCROLL are recorded but not replayed (player movement via
       MOUSE_MOVE is sufficient for rotational fidelity).
     - Multiplayer replay fidelity (other players, chunk hashing, RNG) still stretch.
